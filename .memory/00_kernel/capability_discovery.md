# 能力发现协议

当用户问"你能做什么"、"你有什么能力"、"你擅长什么"或类似问题时，执行以下流程来生成实时的能力介绍。不要依赖任何固化清单，每次都重新执行发现。

## 执行步骤

### 第1步：扫描记忆系统

读取 `.memory/` 下所有目录中的 `.md` 文件，重点关注：

- `entities/` — 找出所有已接入的外部工具、API、数据源
- `preferences/` — 找出用户沉淀的方法论和工作偏好
- `principles/` — 找出系统级的工作规则和约定
- `corrections/` — 找出经过验证的经验教训

### 第2步：分类整理

将扫描结果按"用户能感知到的行动"分类，不按技术组件分类。

分类维度：
- **信息查询类** — 能查什么信息（航班、天气、路线、地理编码等）
- **文档生成类** — 能产出什么文件（报告、表格、演示文稿等）
- **数据处理类** — 能做什么计算和分析
- **开发类** — 能构建什么应用或工具
- **知识积累类** — 有哪些经过验证的方法论可以复用

### 第3步：探索日常工具包

读取 `lab/_toolkit/user_guide.md` 了解工具集全貌。

探索方法：读取 user_guide.md，但对其中提到的每个工具，必须验证对应的脚本文件在 `lab/_toolkit/` 目录下实际存在。user_guide 是从本地 routine 项目净化过来的，可能包含在当前环境中不存在的工具描述。只纳入经过文件存在性验证的能力。

验证方式：列出 `lab/_toolkit/` 及其子目录的实际文件列表，与 user_guide 中描述的工具逐一核对，排除有文档但无文件的工具。

### 第4步：探索当前平台技能

读取当前宿主平台提供的技能/插件列表，了解平台额外提供的能力。

探索方法：查看当前平台的技能注册表或可用工具列表，遍历每个条目的描述字段。注意这些能力是平台绑定的，换平台后可能不同，不要固化具体列表到本协议中。

### 第5步：组装回答

用自然语言组织输出，遵循以下原则：
- 用"你能做什么"而非"我有什么工具"来描述
- 每个能力一句话说清，不给技术细节
- 按使用频率或实用程度排序
- 如果用户追问某项能力的细节，再去读取对应的记忆文件获取具体参数和用法

## 触发条件

- 用户明确询问能力范围
- 用户说"你能做什么"、"你有什么用"、"介绍一下你自己"
- 新对话的首次互动（如果用户表现出不了解你的能力）

## 注意

- 不要展示API Key、内部路径等技术细节
- 不要展示记忆文件的文件名和目录结构
- 能力介绍的目标是让用户知道可以找你做什么，而不是展示技术栈
