# 最小旅行单元判定框架

## 用途

将省级区域分解为离散的独立旅行单元。每个单元为一个行政单位（地级市、县级市、县、区），旅行者可以独立做出"我要去XX"的出行决策。

本框架仅用于旅行规划的第一阶段：**省 → 旅行单元**。不适用于单个景点或景区。

## 定义

最小旅行单元 = 旅行者可以独立做出"我要去XX"决策的地方。

## 判定流程

按顺序依次检验以下三个维度：

### 维度1：住宿检验

这个地方有没有自己的住宿生态（酒店、民宿、客栈）？

- 有 → 进入维度2
- 没有 → **非独立**，结束

### 维度2：时长检验

这个地方的旅行内容能不能撑满至少一整天？

- 能 → **独立**，结束
- 不能 → 进入维度3

### 维度3：当天往返可行性

分两种情况，均在维度2不通过时触发。

**情况A：地级市以下行政单位**

假设从所属地级市核心区当天往返，计算有效游玩时间：

1. 扣除往返路途时间
2. 扣除两顿正餐时间（约1.5小时）
3. 剩余时间 = 有效游玩时间
4. 对比完成该地方核心玩法所需的最低时间

- 有效时间不够 → **独立**，结束
- 有效时间够用 → **非独立**，结束

**情况B：同级相邻吸收**

该单位为地级市或以上级别，但维度2不通过（体验不够一天），检查附近是否存在同时满足以下全部条件的相邻目的地：

1. 自身维度2通过（体验足够丰富）
2. 当天往返轻松（单程不超过约1小时）
3. 旅行体验类型兼容（不会出现沙漠直接跳到海洋等突兀切换）

- 存在这样的吸收者 → **非独立**，被吸收
- 不存在 → **独立**（虽然薄，但周围没有能吸收它的地方）

## 关键设计原则

1. **维度3是单向的** — 维度3只有"倾向独立"的判定力，没有"倾向合并"的判定力。当有效时间够用时，维度3不表态。合并的结论只能来自维度1不通过，或情况B的吸收机制。

2. **仅在维度2失败时触发吸收** — 只有体验不够厚的地方才会被判定是否需要被吸收。无需穷举所有相邻单位，只需看最近的、体验更丰富的那一个。维度2通过的地方，不管旁边有谁，都是独立的。

3. **适用范围** — 本框架仅用于行政单位的省级分解。景点、景区、自然地标等由后续下游流程处理。

## 在旅行规划管线中的位置

本框架是旅行规划自动化管线第一阶段：

```
输入：一个省
  ↓ 遍历该省所有行政单位（地级市+下辖县/县级市/区）
  ↓ 逐个跑本框架（维度1住宿→维度2时长→维度3当天往返）
输出：该省的独立旅行单元列表
  ↓ 第二阶段（独立流程）
  ↓ 对每个旅行单元：研究内部景点 + 以交通枢纽为原点计算驾车距离/时间
输出：每个旅行单元的景点与交通数据
```

第二阶段的具体流程另案定义。
