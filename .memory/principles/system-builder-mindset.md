# 系统构建者思维原则

## 核心规律

### 不满足于表面解决
- 遇到问题时，不满足于"能用"，追求"理解为什么能用"
- 解决问题的过程中，顺手把底层系统也构建了
- 工具产出往往是解决问题的副产品，而非最初目的

### 知识重构而非知识消费
- 深度学习时不是"读过就过"，而是用自己的框架重新组织
- 输出形式可能是长篇笔记、多维分析、或系统原型
- 重构过程本身就是理解深化的方式

### 框架优先于结论
- 面对复杂议题，倾向绘制多视角地图而非得出单一结论
- 框架的价值在于提供导航，不在于给出答案
- 允许框架本身在理解深化后迭代更新

## Agent 协作启示

### 协作模式
- Victor 提供框架和方向，Agent 填充细节和执行
- 当 Victor 完成框架搭建后，信任 Agent 自主运行，不过度干预

### 任务设计建议
- 不只是解决表面问题，考虑是否需要同时构建底层系统
- 长期任务应留下可复用的结构，而非一次性输出
- 当发现规律时，主动提出是否需要固化成模板或协议

## 识别信号

当 Victor 说以下内容时，可能是系统构建者思维在运作：
- "这个问题背后可能有个更通用的模式"
- "顺便把 XX 也做了/整理了"
- "我们来画一下这个的框架"
- "这不是 X 的问题，是整个系统的问题"
